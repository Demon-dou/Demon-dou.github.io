发表于2023-04-15|更新于2023-04-17|Java
|阅读量:

Redis简介

首先，Redis是一个数据库，与传统的关系型数据库不同的是，redis的数据存在于内存中，读写速度快，主要应用方向有缓存，分布式锁等，redis支持事务，持久化，集群等。

为什么要用Redis/为什么要用缓存

从高性能和高并发两个角度看待这个问题

高性能：
假设用户第一次访问数据库，获取数据的速度较慢，因为是从硬盘中读取的，将读取到的数据存入到缓存中，下次访问的时候就可以直接从缓存中获取了，因为是从内存中读取，性能提高了，若数据库中的数据发生改变，要同步改变缓存中的相应数据

高并发：
走缓存的能承受的请求数是远远大于直接操作数据库的，可以考虑将数据库中的某些数据存入到缓存中，这样用户请求的时候直接走缓存而不用走数据库

为什么使用Redis而不用map/guava做缓存？

  1. 本地缓存
  2. 分布式缓存

map/guava实现的是本地缓存，生命周期随着JVM的销毁而结束，多实例的情况下，每个实例要各自保存一份缓存，缓存不具有一致性
Redis实现的是分布式缓存，多实例情况下，共用一份缓存，缓存就有一致性，但相对于map来说，程序架构较为复杂

Redis的线程模型

Redis内部使用文件事件处理器，它是单线程的，所以Redis又称为单线程模型
它采用IO多路复用机制去监听多个socket，将socket产生的事件放入队列中，把事件交给对应的事件处理器进行处理

Redis和Memcached区别

  1. Redis拥有多种数据类型，可以实现多种应用场景，而Memcached只支持简单的数据类型

  2. Redis是单线程的IO多路复用，Memcached是多线程，非阻塞IO复用

  3. Redis支持持久化，可以将数据写入到磁盘，Memcached数据只存在与内存

  4. Redis有原生集群，Memcached只能依靠客户端实现集群分片

Redis常见数据结构及其使用场景分析

  1. String
    String数据结构是简单的key-value类型。使用场景：用来计数（微博数，粉丝数…），用作key-value缓存
  2. Hash
    提供映射表，使用场景：适用于存储对象，后续操作时，可以修改对象中某个子弹的值，如：存储用户信息，商品信息…
  3. List
    链表，使用场景：微博关注列表，消息列表…
  4. Set
    与List类似，但Set可以自动排重。应用场景：实现共同关注，共同粉丝等（将一个用户的关注人存在一个集合，其所有粉丝存在一个集合，交集的过程）
  5. Sorted Set
    可以实现有序排列，应用场景：如直播间的在线用户列表，礼物排行榜

Redis的key达到过期时间后，Redis是如何删除这种key的

定期删除 + 惰性删除

定期删除：
Redis会每隔100ms去随机抽取一些设置了过期时间的key，去检查这些key是否过期，若过期了就删除。随机抽取的原因：若不随机，将所有的设置了过期时间的key都取出来查询的话，会给CPU带来很大的负载

惰性删除：
假设有过期了的key，系统主动的去查一下那个key，若过期了，Redis会给删除掉

若两者都没删除掉过期的key，会导致内存中存在大量的过期key，这时候就引申到了Redis的内存淘汰机制

Redis的内存淘汰机制

MySQL中有2000w数据，Redis中只存20w的数据，如何保证Redis中的数据都是热点数据？

Redis提供了6中数据淘汰策略，在4.0后有增加了两种

  1. volatile-lru: 从已设置过期时间的数据集中，挑选最近最少使用的数据淘汰

  2. volatile-ttl: 从已设置过期时间的数据集中，挑选将要过期的数据淘汰

  3. volatile-random： 从已设置过期时间的数据集中，任意选择数据淘汰

  4. allkeys-lru: 当内存不足以写入新数据时，在键空间中，移除最近最少使用的key（最常用的）

  5. allkeys-random： 从数据集中任意选择数据淘汰

  6. no-eviction: 禁止淘汰数据，当内存不足的时候，新写入操作会报错

  7. volatile-lfu: 从已设置过期时间的数据集中，挑选最不经常使用的数据淘汰

  8. allkeys-lfu: 当内存不足以写入新数据时，在键空间中，移除最不经常使用的key
