  1. 因为查询的逻辑没有使用到索引
  2. 已经建立过了索引，但是索引失效了，比如：

若是因为没有使用到索引，可以考虑进行建立索引，但建立索引前要考虑到业务逻辑，要建立的索引字段是否会有较多的删改，若有较多的删改，要考虑更换索引

为什么字段的删改会导致索引的效率变低？

  • 索引是根据值建立的，删改多会重建索引

若是因为索引失效了，查看业务逻辑与现有的索引产生冲突的原因，进而考虑优化一下业务逻辑，或者更换索引

什么是mysql

一种关系型数据库，是开源的，性能稳定，默认端口是3306

MyISAM与InnoDB的区别

5.5版本之前的默认数据库存储引擎是MyISAM，5.5之后默认存储引擎是InnoDB，InnoDB支持事务

区别：

  1. 是否支持行级锁，InnoDB支持行级锁和表级锁，默认是行级锁，MyISAM只支持表级锁
  2. 是否支持事务和崩溃后的安全恢复：InnoDB支持，MyISAM强调的是性能
  3. 是否支持外键：InnoDB支持，MyISAM不支持

一般都需要考虑到
项目的后期拓展，事务支持，可并发能力以及崩溃后的安全修复等问题，所以一般我们都选择InnoDB

索引

数据结构

  1. B+树
  2. 哈希索引

一般只有大部分是单条记录查询的时候使用哈希索引，其余大部分情况都使用B+树索引，建议选择B+树

针对于不同的存储引擎，索引底层的实现方式不同

  1. 针对于MyISAM
    B+树的叶节点的data域存放的是数据的地址，在检索索引的时候，根据key找到对于的data域。然后根据data域中的地址，最终找到需要的数据记录。称之为“非聚簇索引”

  2. 针对于InnoDB
    分为主索引和辅助索引
    对于主索引，B+树的叶节点data域存放的完整的数据记录，（主索引是根据数据表的主键（key）建立的）
    对于辅助索引，B+数的叶节点data域存放的是主键的值

在进行主索引搜索时，直接找到key所在的节点即可获得数据
在进行辅助索引搜索时，需要先取出主键的值，然后再找一遍主索引，最后获取到数据记录

事务

逻辑上的一组操作，要么都执行，要么都不执行

典型例子：转账。

四大特性：

  1. 原子性：要么全部完成，要么不起作用
  2. 一致性：执行事务之后，数据要保持一致，多个事务对同一个数据读取的结果是相同的
  3. 隔离性：并发访问数据库的时候，一个用户的事务不能被其他的事务所干扰，各个并发事务之间是相互独立的
  4. 持久性：一个事务被提交后，对数据库的数据改变是永久的，即使数据库发生故障也不会受到影响
并发事务会产生的问题
  1. 脏读： 一个事务访问数据并对其进行修改，但还没有提交到数据库中，此时，另一个事务访问数据，得到的就是脏数据
  2. 丢失修改： 两个事务都在访问数据，第一个事务将数据修改后，第二个事务也进行修改，那第一个事务修改的数据就丢失了
  3. 不可重复读： 一个事务内对数据进行多次读取，此时另一个事务也读取该数据，并对其进行了修改，那第一个事务对数据读取的结果就会不同
  4. 幻读： 一个事务在读取一些数据，另一个事务对插入了一些数据，第一个事务获取到原本不存在的数据

不可重复读与幻读的区别：不可重复读—针对于改， 幻读—针对于增删

事务的隔离级别？MySQL的默认隔离级别？

四种隔离级别：

  1. 读取未提交
    最低的隔离级别，允许读取未提交的数据变更，可能导致：脏读，幻读，不可重复读
  2. 读取已提交
    读取并发事务已提交的数据，可避免脏读，可能导致：幻读，不可重复读
  3. 可重复读
    对同一个字段的多次读取结果一致，除非数据被本身事务进行修改，可避免脏读和不可重复读，可能导致：幻读
  4. 可串行化
    最高的隔离级别，完全服从ACID，所有事务依次执行，完全不干扰，可防止脏读，幻读，不可重复读

MySQL的默认隔离级别： 可重复读

SQL在InnoDB的可重复度的事务隔离级别下使用的是Next-key Lock锁算法，可避免幻读的产生，达到了可串行化的隔离级别

表级锁与行级锁

表级锁：对整张表加锁，实现简单，资源消耗少，不会出现死锁，MyISAM和InnoDB都支持
行级锁：对当前操作的行加锁，可大大减少数据库操作的冲突，并发度高，开销大，可能出现死锁

MySQL的大表优化

当单表记录数过大时，数据库的CRUD性能会下降，优化措施：

  1. 限定数据的范围：禁止不带任何限制的去查询所有数据的语句
  2. 读写分离：主库负责写，从库负责读
  3. 垂直分区：根据数据表的列进行拆分，把一张表拆分成多张表
    优点：可简化表的结构，易于维护
    缺点：主键会出现冗余，引起join操作
  4. 水平分区：保证表的结构不变，按照某种策略进行数据分片，将数据分散到不同的表或不同的库中（最好分库，可分布到不同机器上，更有利于提升性能）

数据库分片的两种常见方案

  1. 客户端代理：分片逻辑在应用端，封装在jar包，通过修改或封装JDBC层实现。如：Sharding-JDBC,TDDL

  2. 中间件代理：在应用和数据中间加一个代理层，分片逻辑统一维护在中间件服务。如：Mycat

分库分表后的id处理方式？

通过设置全局唯一的id来支持： 美团的Leaf分布式ID生成系统

几种考虑情况

  1. UUID
  2. 数据库自增id
  3. Redis生成id
  4. Twitter的snowflake算法
  5. 美团的Leaf分布式ID生成系统（推荐）

池化设计思想，数据库连接池？为什么需要数据库连接池？

池化设计思想：初始预设资源，可以避免每次获取资源消耗

数据库连接池：提前创建数据库连接，将其放入到连接池中，方便重用，减少单独连接数据库的时间

为什么需要数据库连接池？

  1. 创建数据库连接后，将其放入到连接池中，再次使用的时候，不需要建立新的连接，若池中现有连接有全部占用，会再次建立一个新的连接放入到池中。
  2. 减少了用户等待建立与数据库连接的时间

一条SQL语句在mysql中如何执行的

参考：一条SQL语句在MySQL中如何执行的

MySQL的基础架构
分为server层和存储引擎层

server层： 所有跨存储引擎的功能都在这一层实现，如：存储过程、触发器等，还有一个binlog日志模块

  1. 连接器：负责身份认证和权限相关（登录MySQL时）
  2. 查询缓存：执行查询语句的时候，先查询缓存（MySQL8.0后移除）
  3. 分析器：没有命中缓存的话，SQL语句会经过分析器，词法分析：分析SQL要干什么，语法分析：语法是否正确
  4. 优化器：按MySQL认为最优的方案进行执行
  5. 执行器：执行语句，校验用户是否有权限，若有权限，然后从存储引擎返回数据

存储引擎层： 主要负责数据的存储和读取，采用可替换的插件式架构，InnoDB中自有redolog日志模块

查询和更新

对于查询：
通过连接器进行身份认证和权限判断，若没有权限返回错误信息，否则，去查询缓存，若内存中存在value，则直接返回结果，否则，在分析器进行词法分析，将SQL语句按照关键词进行拆分，去判断SQL语句具体要进行什么操作，然后进行语法分析，判断SQL语句是否有语法错误，接着，进入到优化器进行优化，选择优化器认为最有的方案，进入到执行器，优先判断是否有权限，若没有权限，返回错误信息，否则，调用存储引擎接口，返回结果。

对于更新：需要应用到两个日志：binlog和redolog
先通过分析器分析SQL语句的功能以及语法，通过权限校验之后，使用执行器去执行，调用引擎接口，修改数据，存入内存，记录到redolog中，更新状态为prepare，然后告诉执行器，执行完成了，随时可以提交，执行器收到通知后记录binlog，然后调用引擎接口，提交redolog为提交状态，更新完成。

在redolog的状态为prepare，binlog写完了，发生异常重启的时候，
会先判断redolog是否完整，完整立即提交，若redolog只是commit状态，这时去判断binlog是否完整，完整，提交redolog，否则，回滚事务。 由此来解决数据一致性的问题

MySQL高性能优化规范建议

参考：MySQL高性能优化规范建议,速度收藏

数据库命名规范

  1. 小写字母且用下划线分割
  2. 禁止使用mysql保留的关键字
  3. 命名不超过32个字符
  4. 临时表使用tmp_为前缀，日期为后缀，备份表使用bak_为前缀，日期为后缀
  5. 存储相同数据的列名和列类型保持一致（作为关联列）

数据库基本设计规范

  1. 数据库和表的字符集统一使用utf-8, 避免字符集转换，若有存储emoji表情，字符集采用utf8mb4
  2. 所有表和字段都需要添加注释
  3. 尽量控制单表数据量的大小，控制在500万以内
  4. 禁止在数据库中存储图片，文件等大的二进制数据
  5. 禁止在线上做数据库压力测试
  6. 禁止从开发环境，测试环境直接连接生产环境

数据库字段设计规范

  1. 优先选择符合存储需要最小的数据类型
  2. 避免使用TEXT, BLOB数据类型，最常见TEXT类型可以存储64k数据
  3. TEXT或BLOB类型只能使用前缀索引，TEXT列不能有默认值
  4. 避免使用ENUM类型
  5. 尽可能把所有列定义为NOT NULL
  6. 使用TIMESTAMP（4个字节）或DATETIME（8个字节）存储时间
    注: 使用字符串存储日期型数据，会导致无法用日期函数进行计算和比较，且占用更多的空间
  7. 跟金额类型的数据相关的必须使用decimal类型

索引设计规范

  1. 限制每张表上的索引数量，建议单张表索引不超过5个
  2. 禁止给表中的每一列都建立单独的索引
  3. 常见索引列建议：
    • 出现在select、update、delete语句的where从句中的列
    • 包含在order by、group by、distinct中的字段
    • 不要将符合1,2中的字段的列都建立一个索引，通常将1、2中的字段建立联合索引效果更好
  4. 多表join的关联列
  5. 如何选择索引列的顺序
    1. 区分度最高的放在联合索引的最左侧
    2. 尽量将字段长度小的列放在联合索引的最左侧
    3. 使用最频繁的列放在索引的最左侧
  6. 避免建立冗余索引和重复索引（会增加优化器生成执行计划的时间）
  7. 对于频繁的查询优先考虑使用覆盖索引（包含了所有查询字段的索引（包括where，select，orderby, groupby））
  8. 索引SET规范
    • 不建议使用外键约束，但要在表与表中的关联键上建立索引

数据库SQL开发规范

  1. 避免数据类型的隐式转换

    select name, phone from customer where id = '123';

  2. 充分利用表上已经存在的索引

  3. 数据库设计时，应该对以后扩展进行考虑

  4. 禁止使用select * ，必须使用select <字段列表> 查询

  5. 禁止使用不含字段列表的INSERT语句

  6. 避免使用子查询，可将子查询优化为join操作（子查询的结果集无法使用索引）

  7. 避免使用join连接太多的表（建议不超过5个）

  8. 尽可能的少使用or

  9. 拆分复杂的大SQL为多个小SQL

数据库操作行为规范

  1. 超100万行的批量写（update，insert，delete）操作，要分批多次进行操作

一条SQL语句执行很慢的原因？

分为两种情况

  1. 大多数情况正常，只是偶尔会出现很慢的情况
  2. 在数据量不变的情况下，这条SQL语句一直以来都执行的很慢

针对于偶尔很慢（与SQL语句本身没有关系）

  1. 数据库在刷新脏页（内存与磁盘的数据页内容不一致）
  2. 拿不到锁

针对于一直都很慢

  1. 没有用上索引
    （1）字段没有索引
    （2）字段有索引，但没有用到
    （3）函数操作导致没有用上索引
  2. 数据库自己选错索引了（若判断没有走应该走的索引，可以通过强制走索引的方式来查询）
