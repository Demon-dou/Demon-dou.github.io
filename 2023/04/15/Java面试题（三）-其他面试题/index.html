日活与月活

日活：一天之内打开某产品的用户数(去重后)
月活：一个月的时间内打开某产品的用户数(去重后)

日活/月活：结果是一个数字，在1~1/30之间，数值越大，说明用户越忠诚，但也受到产品本身的特征，以及是否是新用户爆发期等因素影响

乐观锁和悲观锁

乐观锁

总是假设最好的情况，默认别人不会修改数据，读取数据时不用加锁直接读取，但在更新的时候要判断一下在此期间是否有人更新了这个数据

实现方式：

  1. 版本号机制
  2. CAS算法

例子：类似于数据库的write_condition机制

适用场景：多读的场景，可以提高吞吐量

缺点（CAS的）：（解决办法暂时不太明了）

  • ABA问题
  • 循环时间长开销大
  • 只能保证一个共享变量的原子操作（可以通过Atomicference类，将多个要共享的变量合并成一个）

悲观锁

总是假设最坏的情况，默认别人会修改数据，所以每次拿数据的时候都会进行加锁

例子：

  1. 传统的关系型数据库里的行锁，表锁，读锁，写锁…
  2. Java中的synchronized和ReentrantLock

适用场景：多写的场景

版本号机制

在数据表中增加一个版本号字段（version）,表示数据被修改的次数，当数据被修改的时候，version + 1

  1. 先查询数据（带有version）
  2. 更新时，通过where判断数据库中的version和查询出来的version是否相等
  3. 若相同则更新数据，并将版本号加一
  4. 若不相同则表示该数据发生了并发，被别的线程使用了，则再次尝试更新（递归），知道成功更新数据为止
select version from test;  (1)

update test set a + 12, version + 1 where version = ${(1)}

CAS算法

即Compare and swap（比较与交换），一种有名的无锁算法，在无锁的情况下，也能实现多线程之间的变量同步。CAS算法主要涉及三个操作数

  • 需要读写的内存值V
  • 进行比较的值A
  • 拟写入的新值B

当且仅当V的值等于A时，CAS通过原子方式用新值B来更新V的值，否则不会执行任何操作（比较和替换是一个原子操作）。一般是一个自旋操作，即不断的重试

自旋锁：当一个线程获取锁的时候，如果锁已经被其他线程获取，那么该线程将循环等待，然后不断判断是否能够成功获取，知道获取到锁才会退出循环

自旋锁与互斥锁:

  • 自旋锁与互斥锁都是为了实现保护资源共享的机制
  • 无论是自旋锁还是互斥锁, 在任一时刻,都最多有一个保持者
  • 获取互斥锁的线程, 若锁已被占用，则线程进入睡眠状态；获取自旋锁的线程是一致循环等待锁释放（线程状态不变）
